# What is Blobcade?

Blobcade is a **free, open-source browser game platform & engine**: friendly blocky
worlds with AAA-style lighting. Type one command, get a portal of multiplayer games,
and create your own in ~25 lines of TypeScript.

It is **not** a general-purpose engine like Unity, Unreal, or Godot, and it does **not**
use any proprietary engine. The renderer is built entirely on [three.js](https://threejs.org/) (MIT).
It runs in any modern browser with no downloads, no accounts, and no asset pipeline —
every model, texture-free material, sound, map and name is original and generated by
code at runtime.

## The core idea

Blobcade is a **genre engine + platform seed**. Within its genre family — blocky
platformers/obbies, voxel sandboxes, arena shooters, CTF, battle royale, adventure
maps — a creator gets a complete multiplayer-flavored game from one TypeScript file,
and can extend the engine itself through registries, events and systems.

The design is reverse-engineered from **why** the biggest blocky game platforms work:

| Insight | How Blobcade implements it |
| --- | --- |
| Third-person **WASD + Space**, drag-to-orbit camera, scroll zoom, shift-lock, **R to reset** | The default control scheme |
| The **obby** (obstacle course) is blocky gaming's foundational genre | *Sky Obby* — 15 stages, checkpoints, kill bricks, spinners, ice, bounce pads |
| Blocky avatars with simple faces — simplicity is identity | Procedural rounded-box avatar, canvas-drawn smile, per-player shirt colors |
| Social presence drives retention | WebSocket rooms, interpolated avatars, chat bubbles, join/leave toasts |
| Creation must be radically easy | `defineGame({...})` SDK — a complete game is one small file |
| A **portal of games**, not one game | Hash-routed portal with game cards; adding a game = 1 import |
| Play → earn → spend keeps creators and players invested | Session coin economy + Blobcash wallet |

## The "looks expensive" part

The bet is that **premium light on friendly blocky shapes** beats realistic geometry.
Low-poly worlds + a modern post-processing stack = runs everywhere, looks expensive:

- **ACES filmic tone mapping** — the industry-standard filmic curve
- **Bloom** — via three.js's `UnrealBloomPass` addon (lava, neon and sun glow)
- **GTAO** ground-truth ambient occlusion pass
- **SSR** screen-space ray-traced reflections on shiny materials (ice, metal, gold, glass, water)
- PCF soft shadows snapped to texels (no shimmer), following the player
- Sky-tinted fill light + boosted hemisphere as fake GI, so shadows never crush to black
- Procedural sky shader: atmosphere gradient, sun disc + halo, FBM clouds, night stars

## Three ways to create

Blobcade offers three deliberate creation paths, trading power for trust — but they're
**one ladder, not silos**: a TypeScript game can load an ASCII map or start from a doc, a
GameDoc can mix a map + parts + rules + a sandboxed script, and the Studio can export a doc
straight to TypeScript. **[How the paths relate →](./creation-paths.md)**

1. **Pure TypeScript** (trusted) — the full-power `defineGame()` SDK, the same way the
   six shipped games are built. Full imports, arbitrary state, `onTick`, bots,
   `ctx.engine` access. See [Pure TypeScript games](./tutorials/typescript-game.md).
2. **Map / GameDoc editor** (community-safe, no-code) — visual builders and ASCII maps
   that produce a portable **GameDoc** (a single JSON document), with declarative
   `when / if / do` rules for logic. See the [map tutorials](./tutorials/text-maps).
3. **Sandboxed scripting** (community-safe, code) — the escape hatch for logic that rules
   can't express (loops, arithmetic, spawners), running in a locked-down Web Worker. It
   lives *inside* a GameDoc's `script` field. See [Scripting](./tutorials/scripting.md).

## Honest scope

Current ceilings, so you know before you build:

- **Multiplayer is client-heavy**: rooms, presence, chat, relayed game events, voxel
  co-build sync and plausibility-capped PvP hit arbitration exist, but transforms and
  most game logic remain client/host driven. Server-authoritative sessions are the #1
  roadmap item.
- **Code games live in this repo**: adding a first-party game = adding a file to
  `src/games/`. There is no runtime upload/publish path for *full game logic* yet (the
  editors cover maps and data docs).
- **Physics is axis-aligned boxes** — no slopes, rotated colliders, vehicles-physics-on-ragdoll.
- **Procedural-only content** — no model/asset import (that's also the zero-pipeline appeal).
- Character-controller games are the sweet spot; top-down/2D/card genres aren't served.
- localStorage persistence for wallet/inventory/drafts; no accounts or cross-device saves yet.
